# Payment Model Documentation

## Overview

The Payment model manages all payment transactions in the MERN Education Platform. It supports multiple payment methods including international providers (Stripe, PayPal) and Ethiopian payment systems (Telebirr, M-PESA, CBE Birr, CBE, Awash Bank, Siinqee Bank). The model tracks payment status, stores transaction metadata, and enforces security requirements such as never storing credit card details.

## Schema Structure

### Main Fields

| Field | Type | Required | Description |
| :---- | :--- | :------- | :---------- |
| `userId` | ObjectId | Yes | Reference to the User making the payment |
| `courseId` | ObjectId | Yes | Reference to the Course being purchased |
| `amount` | Number | Yes | Payment amount (must be > 0) |
| `currency` | String | Yes | ISO 4217 currency code (USD, EUR, ETB) |
| `paymentMethod` | String | Yes | Payment method used (see Payment Methods) |
| `status` | String | Yes | Payment status (defaults to PENDING) |
| `transactionId` | String | Yes | Unique transaction identifier |
| `metadata` | Object | Yes | Payment metadata (see Metadata section) |
| `completedAt` | Date | No | Timestamp when payment was completed |
| `refundedAt` | Date | No | Timestamp when payment was refunded |
| `createdAt` | Date | Auto | Timestamp when payment was created |
| `updatedAt` | Date | Auto | Timestamp when payment was last updated |

### Payment Methods

The model supports the following payment methods:

**International Methods:**

- `CREDIT_CARD` - Credit card payment
- `DEBIT_CARD` - Debit card payment
- `PAYPAL` - PayPal payment
- `STRIPE` - Stripe payment gateway

**Ethiopian Payment Methods:**

- `TELEBIRR` - Telebirr mobile payment (requires phone number)
- `MPESA` - M-PESA Ethiopia mobile payment (requires phone number)
- `CBE_BIRR` - CBE Birr mobile payment (requires phone number)
- `CBE` - Commercial Bank of Ethiopia
- `AWASH_BANK` - Awash Bank
- `SIINQEE_BANK` - Siinqee Bank

### Payment Status

| Status | Description |
| :----- | :---------- |
| `PENDING` | Payment initiated but not completed (default) |
| `COMPLETED` | Payment successfully processed |
| `FAILED` | Payment processing failed |
| `REFUNDED` | Payment was refunded to the user |

### Supported Currencies

- `USD` - United States Dollar
- `EUR` - Euro
- `ETB` - Ethiopian Birr

### Metadata Subdocument

The `metadata` field stores additional information about the payment:

| Field | Type | Required | Description |
| :----- | :---- | :------- | :---------- |
| `ipAddress` | String | Yes | IP address of the user (IPv4 or IPv6) |
| `userAgent` | String | Yes | Browser user agent string |
| `phoneNumber` | String | Conditional | Ethiopian phone number (+251XXXXXXXXX) - required for Telebirr, M-PESA, and CBE Birr |
| `gatewayResponse` | Mixed | No | Raw response from payment gateway |

## Validation Rules

### Amount Validation

- Must be a positive number greater than 0
- No maximum limit enforced at model level
- Should match course price (enforced at business logic level)

### Currency Validation

- Must be one of: USD, EUR, ETB
- Automatically converted to uppercase
- ISO 4217 format

### Transaction ID

- Must be unique across all payments
- Required for all payments
- Typically generated by payment gateway

### IP Address Validation

- Supports both IPv4 and IPv6 formats
- IPv4 format: `xxx.xxx.xxx.xxx` (0-255 for each octet)
- IPv6 format: Standard IPv6 notation

### Phone Number Validation (Ethiopian)

- Format: `+251XXXXXXXXX` (country code + 9 digits)
- Required for: Telebirr, MPESA, CBE Birr payment methods
- Optional for other payment methods
- Validated using regex: `/^\+251\d{9}$/`

## Indexes

The model includes the following indexes for query optimization:

1. **userId** (non-unique) - For querying user's payment history
2. **courseId** (non-unique) - For querying course payments
3. **status** (non-unique) - For filtering by payment status
4. **transactionId** (unique) - Ensures transaction uniqueness
5. **createdAt** (descending) - For sorting by date

## Pre-Save Hooks

### Ethiopian Mobile Payment Validation

Before saving a payment, the model validates that Ethiopian mobile payment methods (Telebirr, M-PESA, CBE Birr) include a phone number in the metadata:

```typescript
if (ETHIOPIAN_MOBILE_PAYMENT_METHODS.includes(this.paymentMethod)) {
  if (!this.metadata.phoneNumber) {
    throw new Error('Phone number is required for [method] payment method');
  }
}
```

**Parameters:**

- `phoneNumber` (string) - Phone number to validate

**Returns:**

```typescript
{
  valid: boolean;
  error?: string;
}
```

**Example:**

```typescript

  userId: user._id,
  courseId: course._id,
  amount: 99.99,

  paymentMethod: PaymentMethod.STRIPE,

    gatewayResponse: {
      chargeId: 'ch_123456',
```

## Security Considerations

### Never Store Sensitive Card Data

**CRITICAL:** The Payment model is designed to NEVER store:

- Credit card numbers
- CVV/CVC codes
- Card expiration dates
- Full card details

All card processing should be handled by payment gateways (Stripe, PayPal) using tokenization.

Only store:

- Transaction IDs from the gateway
- Payment status
- Gateway responses (which should not contain sensitive card data)

### PCI DSS Compliance

To maintain PCI DSS compliance:

1. Use payment gateway tokens instead of raw card data
2. Store only transaction IDs and status
3. Never log or expose card details
4. Use HTTPS for all payment requests
5. Implement proper access controls

### Phone Number Privacy

Ethiopian phone numbers are stored for mobile payment methods but should be:

- Encrypted at rest (database-level encryption)
- Protected by access controls
- Only accessible to authorized personnel
- Included in GDPR/privacy compliance measures

## Business Logic Integration

### Payment Verification Before Enrollment

```typescript
// Verify payment is completed before creating enrollment
const payment = await Payment.findById(paymentId);

if (!payment) {
  throw new Error('Payment not found');
}

if (payment.status !== PaymentStatus.COMPLETED) {
  throw new Error('Payment not completed');
}

if (payment.userId.toString() !== studentId.toString()) {
  throw new Error('Payment user mismatch');
}

if (payment.courseId.toString() !== courseId.toString()) {
  throw new Error('Payment course mismatch');
}

// Proceed with enrollment creation
const enrollment = await Enrollment.create({
  studentId,
  courseId,
  paymentId: payment._id
});
```

### Revenue Analytics


## Error Handling

### Common Validation Errors

```typescript
try {
  const payment = await Payment.create(paymentData);
} catch (error) {
  if (error.name === 'ValidationError') {
    // Handle validation errors
    if (error.errors.amount) {
      console.error('Invalid amount:', error.errors.amount.message);
    }
    if (error.errors.currency) {
      console.error('Invalid currency:', error.errors.currency.message);
    }
  } else if (error.code === 11000) {
    // Duplicate transactionId
    console.error('Transaction ID already exists');
  }
}
```

### Pre-Save Hook Errors

## Migration Notes

When migrating existing payment data:

1. Ensure all transactionIds are unique
2. Validate currency codes are in ISO 4217 format
3. Convert phone numbers to +251XXXXXXXXX format
4. Ensure metadata.ipAddress and metadata.userAgent are present
5. Set default status to PENDING if not specified
